# 在庫状況チェックの開発とパフォーマンスに関する考慮事項

在庫の正確性は非常に重要な検討事項です。 バックオーダーや在庫切れのしきい値の設定など、このリスクを可能な限り低くするのに役立つネイティブ機能がいくつかあります。 これらのトピックは両方とも[Experience League](https://experienceleague.adobe.com/en/docs/commerce-admin/inventory/configuration/backorders)で読むことができます。

Adobe Commerceストアでリアルタイムの在庫状況チェックが要求されるプロジェクトやユースケースがあります。 このチュートリアルでは、開発とパフォーマンスに関する考慮事項を含め、この会話を処理するためのinsightについて説明します。

## この要求が絶対に必要かどうかを検証します

できるだけ多くの情報を活用して、会話に参加する準備を整えましょう。 最も重要なことは、ネイティブ機能がこのプロジェクトで受け入れられないことを確認することです。 このリクエストの背後にある理由を見つけて、Adobe Commerceのネイティブ機能がこのリクエストを満たさないことを検証します。

もうひとつの検討事項は、この機能を開発、テスト、保守するためのコストです。 ある関係者が必要だと考えているからといって、それが要件だというわけではありません。 Adobe Commerceのコア機能の外側で在庫検証を行うと、関連するコストが発生します。 そうしたコストは、技術的負債、テストや検証の強化、アーキテクチャに関する使用状況の文書化やサポート文書に伴います。

## 許容できるインベントリ更新の頻度の決定

在庫チェックと、それが3つのアプローチでどのように達成されるかを検討してみてください。 それぞれに利点と限界があります。 また、複雑さが増し、エラー処理に対してより多くのテストと思考が必要になります。 カスタムルートを使用する場合は、責任と考慮事項が追加されていることを忘れないでください。 たとえば、フォールバックプロセス、モニタリング、テスト、トラブルシューティングなどが開発チームに任せられます。 その際、新しいサポートドキュメント、トレーニング、モニタリング機能を導入し、開発チームが機能全体をサポートできるようにすることが重要です。 副次的な影響としては、開発チームがプロセスを完全に所有し、コアのAdobe Commerceアプリケーションが提供するネイティブ機能を活用していないことがあります。 Adobe サポートは、このレベルのカスタマイズを支援できません。

まず、ネイティブ機能を利用します。 ネイティブ機能を利用することで、リスクを最小限に抑え、多くのメリットを享受できます。 このアプローチを採用すると、Adobe Commerceが提供するすべての既存のドキュメントとチュートリアルを利用して、この機能を使用できます。 在庫管理には多くの側面があるため、アプリケーションに付属しているものを使用することが最初に検討する必要があります。 ただし、注文時にコマースで見つかったデータが完全に正確ではない場合があります。 物事が同期しなくなる可能性のある例としては、注文管理システムで直接Adobe Commerce アプリケーションの外で販売が許可されていることです。 その理由は、正確な在庫量をAdobe Commerceで表すためには、Adobe Commerce情報を可能な限り正確に保つために、何らかの統合が必要になるからです。 過剰販売が許されない場合は、在庫切れのしきい値を追加することは、ゼロになる前に商品の販売を停止するための優れた方法です。 Adobe Commerceのネイティブ同期機能は、1日あたり最大1回です。 これは、いくつかのユースケースには十分ですが、他のユースケースには十分な頻度ではないかもしれません。 詳しくは、[ スケジュールされた読み込みと書き出し](https://experienceleague.adobe.com/en/docs/commerce-admin/systems/data-transfer/data-scheduled-import-export)を参照してください。

2つ目のアプローチは`near real-time`です。 ほぼリアルタイムでネイティブ機能を使用しています。 しかし、これには、コマースを頻繁にフィードし、スケジュールどおりに在庫を更新するための統合を提供するための追加作業が含まれます。 例えば、毎時間。 このオプションでは、統合の仕組みについて少し考える必要がありますが、「bulk api」を使用し、いくつかのミドルウェアでデータの変換を行い、それをコマースにプッシュすることは素晴らしいアプローチです。 Adobe App Builderなどの基盤を利用して、作業の大部分を処理し、より頻繁にAdobe Commerceにプッシュします。

リスクと責任が最も大きい3つ目のアプローチで、最も複雑なのは、外部APIまたはデータソースに対するリアルタイムのライブインベントリチェックです。 外部システムにリアルタイムで在庫チェックを実施するのはリスクが高く、考慮すべき他の要素もいくつか存在します。 ここでは、その他に評価が必要な項目をいくつか紹介します。

* 外部システムはRESTまたはGraphQL リクエストを受け入れることができます
* エンドポイントには、web サイトトラフィックと一致しない可能性がある1分間あたりのリクエスト数Xなど、制限があります
* ロード中の応答時間はどうなるか
* 応答時間が長い場合はどうなりますか。これを自動的に終了し、ネイティブインベントリなどのフォールバックオプションを使用します。
* API リクエストが許容値制限内であることを確認するために使用できる監視のタイプ

## 非ネイティブ在庫管理を検討する際の考慮事項

カスタマイズは、できるだけ複雑にしないでください。
在庫の組織がどの程度フラットになるのか、それが1つのSKUと利用可能な在庫の合計量であるか、または考慮する必要がある他の属性があります。

在庫情報がskuや利用可能な合計数量など、かなりフラットな場合は、ほぼリアルタイムのオプションが拡張されます。 ほぼリアルタイムという概念は、ソースから在庫を収集し、その後、リクエストに応答するために使用されるストレージエンジンを入力するバックグラウンド操作があることを意味します。 これには、Redis、Mongo、その他の非リレーショナルデータベースなどを使用できます。 これらのオプションは非常に高速で、キーと値のペアに最適なので便利です。 データがもう少し複雑な場合は、コマースアプリケーションの内外でリレーションデータベースを使用する必要があります。 これをコマースデータベースからオフロードすることで、コアコマースアプリケーションをこれらのトランザクションから分離できます。 もう1つの利点は、コマースアプリケーション、CPU、RAMなどのI/Oを使用から救うことです。 Adobe Commerce アプリケーションサーバーのリソースを使用する代わりに、新しいAPIを利用して、オフサイトストレージからデータを取得します。  そのため、あらゆるデータを変換するためのミドルウェアが必要になります。 次に、呼び出し元のアプリケーションが期待どおりに結果を取得できることを確認します。 Adobe App BuilderとAPI Meshを組み合わせて使用すると、データを適切な形式で変換して返すことができます。

API メッシュでAdobe App Builderを使用することも、複数のソースから在庫を収集する場合に最適な方法です。


## 実行ロジックをプロセス外の場所に移動する

Adobe Developer App Builderは、Adobe ソリューションを拡張するためのカスタムエクスペリエンスを統合および作成するための、統一されたサードパーティ拡張性フレームワークを提供します。 Adobe CommerceはAdobe Developer App Builderを使用できます。 これは、通常コアアプリケーションで発生し、オフサイトに移動する機能を拡張するための優れたユースケースです。 Commerce アプリケーションから機能を削除することで、Commerce アプリケーションのモジュール数と複雑さが軽減されます。 これにより、プロセス内でのカスタマイズ数が少なくなり、アップグレードやメンテナンスの複雑さが軽減されます。

Adobeのチームは、インスピレーションを得るためのドキュメントを作成しました。これらのドキュメントは、優れたインスピレーションの源となり、実用的なコードサンプルを提供します。 買い物客が商品をカートに入れた際、サードパーティの在庫管理システムは、その商品の在庫があるかどうかをチェックします。 その場合は、製品を追加します。 それ以外は、エラーメッセージを表示します。  コードサンプルと詳細については、[Webhook ユースケース ](https://developer.adobe.com/commerce/extensibility/webhooks/use-cases/#add-product-to-cart)を参照してください。

## 在庫チェックを実施するタイミング

在庫がまだ利用可能かどうかを確認するのは、最終的にはビジネス関係者、ソフトウェアアーキテクトが他の主要な関係者から何らかの意見を得るまでになります。 適切なタイミングとしては、カートに商品を追加するタイミングやチェックアウトワークフローに入るタイミングなどが考えられます。 その他のイベントは、必要ない場合にバックエンドシステムに負荷を追加するだけです。 在庫の問題を特定する目的は、在庫が最も重要な場合にのみ存在することに留意してください。 その他のチェックは便利かもしれませんが、在庫状況チェックの全体的な目標に影響を与える可能性がある場合は、慎重に検討し、ビジネス関係者や他の人がバックエンドシステムに追加の負荷がかかる可能性があることを認識している場合にのみ許可する必要があります。

## 在庫ソースをどのように調査するか

外部在庫源の包括的な調査が必要です。 評価すべき項目は、使用可能なAPI オプション、GraphQLのサポート、および予想される応答時間です。 インベントリソースの接続帯域幅が制限されている場合、またはリアルタイム要求で使用することを意図していない場合、使用機能は除外され、代わりにほぼリアルタイムを考慮する必要があります。  API リクエスト時間が定義されたパラメーターを超える場合、これは実行可能なオプションからも除外されます。  例えば、APIの応答は1回のリクエストでは200 MS®ですが、中程度の負荷では500～900 MS®に増加します。  読み込みが増え、ライブインベントリの呼び出しが利用できなくなる可能性が高くなり、事態が悪化する可能性があります。

APIの応答時間は、シンプルなリクエストと、ライブ web サイトでの予想トラフィックと同様の大量のリクエストでテストするようにしてください。 現実世界のシナリオをシミュレートするために、コマースのあらゆる領域を同時にテストすることを忘れないでください。 もし期待が商品ページでライブインベントリコールとして発生する場合、カートの表示と編集の際、チェックアウト時に、実際の顧客行動と店舗での期待を模倣するために、同時にこれらすべてをシミュレートする必要があります。

## 代替オプション

在庫ソースがダウンしてモニタリングが利用可能な場合は、Adobe Commerceのネイティブ機能を使用することをお勧めします。 しかし、適切に監視すれば、リアルタイムの在庫チェックの欠如を反映して、顧客体験を動的に変化させることができます。 これは、セールやイベントが早めにキャンセルされたり、過剰な販売を避けるためにディスプレイから削除されたりすることを意味します。 在庫ソースが何らかの理由でダウンした場合に何をすべきか、その期待値について検討し、ストアオーナーと話し合う必要があります。そうすることで、誰もが、そのような不幸な状況で引き継がれる自動プロセスを認識できます。

## まとめ

リアルタイムの在庫チェックの導入は、大きな決断をもたらします。 web サイトオーナーや開発チームなどが十分な教育を受け、あらゆるメリットと潜在的な落とし穴を開発者リードやアーキテクトに周知できるようにします。 慎重なプランを提供して、その理由とフォールバックプロセスを明らかにすることが、成功の鍵となります。

ライブ在庫チェックは可能ですが、QA サイクル中のテストと検証に関する調査と検討が必要です。 負荷テストとエンドツーエンドの自動化されたテストを確実に実施することで、潜在的な問題を捉えて対処できます。

監視によって、外部システムへの失敗した呼び出しの傾向が検出された場合、または応答時間が許容しきい値を超えた場合に実行すべきアクションを検討することで、サイトをオンラインのままにし、顧客に刺激を最小限に抑えることができます。 フォールバックオプションは、外部在庫チェックをオフにしてネイティブ機能を使用するだけで済むような簡単なものや、プロモーションを無効にしたり、エスカレーションする問題を開発チームに通知したり、可能であればリクエストを2番目または3番目のバックエンドシステムに再ルーティングしたりするなど、高度なものにすることができます。 フォールバックメカニズムの実行方法は、すべての統合がある時点で問題を抱えているため、実際の実装と同じように考えるべきです。 自動化されている、または手作業が必要なものは、すべて明確に文書化する必要があります。
